# 26 тАФ The port, the backend and the proxy fallback

The proxy fallback has appeared in every chapter тАФ it is Mockxy's heart. Here we look at
it up close: the complete request-by-request decision, mock-only mode and its "talking"
404s, the proxy's errors and timeouts, and the `x-mock-source` reference table.

## The decision, in full

For every incoming request (excluding the reserved `/_admin/...` paths):

1. **Global controls** тАФ with the server off or in proxy all, no mock, handler or
   middleware intervenes: everything goes straight to the backend (with no backend
   configured: `501`).
2. **CORS preflight** тАФ with automatic CORS on, preflight `OPTIONS` requests get an
   automatic answer (an explicit `OPTIONS` mock takes precedence).
3. **Route match** тАФ chapter 7's path convention: the path first (the most specific
   wins), then the method. If the chosen route defines the method, the selected variant
   answers тАФ mock, handler, or middleware (which still goes through the backend).
4. **No mock** тАФ the **proxy fallback** decides:
   - enabled and backend configured тЖТ forwarded to the real backend;
   - disabled тЖТ **`404 Mock Not Found`**;
   - enabled but no backend тЖТ **`501 Backend Not Configured`**.

## Mock-only mode

Fallback off (or an empty Backend URL) = **mock-only**: what isn't mocked doesn't exist.
The cases where it is the right mode:

- **demos and offline work** тАФ no dependency on external environments: the workspace is
  the entire "API";
- **determinism in tests** тАФ the certainty that the frontend never touches a real
  environment: an unexpected call fails with a 404 instead of producing real effects;
- **development ahead of the backend** тАФ when a backend simply doesn't exist yet.

Mock-only 404s are a diagnostic tool, not a dead end: the body states the reason for the
miss. **`path_not_mocked`** тАФ no route matches: check prefixes (an extra or missing
`/api`) and remember the match covers the whole path. **`method_not_mocked`** тАФ the route
exists but doesn't define that method: and by the two-stage rule there is no falling back
to less specific routes. They are the first two symptoms of the
[troubleshooting chapter](33-troubleshooting.md).

## What gets forwarded

When the request goes to the backend: the URL is the configured base plus path and query
as they are; the headers all pass except the *hop-by-hop* ones (those belonging to the
single leg, like `Connection`), with `Host` rewritten to the backend's; the body travels
**streaming**, unbuffered тАФ large uploads cost no memory. The response comes back with the
original status and headers, plus the cookie, redirect and CORS adaptations of
[chapter 27](27-proxy-topology.md) when active.

## Errors and timeouts

- **Backend unreachable** (connection refused, DNS failure) тЖТ **`502 Bad Gateway`**.
- **Timeout** (`requestTimeoutMs`, default 15 s): covers connection, request send and the
  wait for the **first response headers**; once exceeded, still `502`. Once the response
  has started, the timeout no longer applies: a stream can stay silent for minutes without
  being cut тАФ an upstream dying midway shows up as a connection error.
- **Error after the response has started**: with the headers already sent there is no
  status left to change; the connection is closed abruptly, so the client perceives the
  truncation instead of mistaking it for a complete response.
- **Client walking away**: the leg toward the backend is closed immediately.

For the frontend, the practical rule: a `502` from Mockxy speaks about the **backend**
(down, or slow to *start* answering), a `504` about a local **handler** that didn't answer
in time, a `501` says the backend was needed but isn't configured.

## `x-mock-source`: the reference table

| Value | Meaning |
|---|---|
| `mock` | static response from a mock |
| `handler` | generated by a local handler |
| `middleware` | backend response transformed by a middleware |
| `backend` | proxied to the backend untransformed (including the middleware bypass on streams and oversized responses) |
| `mock-only` | `404` with proxy fallback disabled |
| `backend-unconfigured` | `501` because no backend is configured |
| `cors-preflight` | preflight `OPTIONS` handled by automatic CORS |

The same values populate the monitor's "Served by" column; the combined "Real backend"
filter gathers everything that did not come out of the workspace's mocks and handlers.

The proxy forwards faithfully тАФ but between a browser and a backend designed to be
reached directly, absolute fidelity would break login and navigation. The three
adaptations that prevent it are the [next chapter](27-proxy-topology.md).
