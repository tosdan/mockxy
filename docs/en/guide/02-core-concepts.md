# 02 — Core concepts

Before installing anything, it pays to fix the vocabulary that the interface (and this
guide) uses on every page, and to understand how Mockxy decides, request by request, who
answers. It is a handful of notions, but everything else rests on them.

## The vocabulary

**Workspace.** The folder that contains an entire mock environment: the endpoint
definitions, the reusable data files, the local settings and the captured traffic. It is a
self-contained, portable unit: you version it in git, share it with the team, open it in
the desktop app or serve it with the headless server. A frontend project typically keeps
its workspace inside the repository, next to the code.

**Endpoint.** An HTTP method plus a path, mocked: `GET /api/users/:id` is one endpoint,
`POST /api/users` is another. Each endpoint can be enabled or disabled: disabling it does
not delete it — requests on that route simply go back to flowing toward the real backend
(or to the 404, in mock-only mode).

**Response variant.** The same endpoint can hold several ready responses: the 200 with
data, the empty list, the 404, the 500. At any moment **exactly one variant is selected**,
and it is the one the endpoint serves; changing behavior means selecting another variant,
without touching any content. It is the mechanism by which, in a few seconds, you try how
the frontend reacts to the error case. Variants come in five natures — static mock,
handler, middleware, SSE, WebSocket — introduced in chapter 1 and covered in depth in
parts II and III.

**Collection.** The folder-like organization of the catalog: endpoints are grouped into
collections, nested if needed, to keep a growing workspace tidy. It is interface metadata:
moving an endpoint into a collection changes neither the served path nor the position of
the files on disk.

**Monitor.** The real-time view of the traffic crossing Mockxy — mocked and proxied — with
the ability to turn what passes by into mocks. Together with the proxy it is the
application's other pillar (chapters 20–22).

## The principle: the interface and the files are the same thing

Every action in the interface writes plain JSON files inside the workspace, and every
hand-made change to the files shows up in the interface thanks to hot reload. There is no
database, no hidden state: **the workspace is the source of truth**, and the interface and
the files are two equivalent views over the same data.

The practical consequences matter: the whole mock set is versioned in git and travels with
the project; a mass refactoring can be done with a search-and-replace in your editor; and
nothing you do from the interface is opaque — you can always open the file and see what is
written there. The file structure is the subject of [chapter 24](24-mocks-as-files.md).

## The journey of a request

When a request reaches Mockxy (excluding the reserved `/_admin/...` paths, used by the
interface itself), the decision about who answers always follows the same order:

1. **Global controls.** Two switches in the interface's top bar govern the whole engine:
   with the mock server **off**, or in **proxy all** mode, no mock intervenes and
   everything goes straight to the real backend. They exist to suspend mocking wholesale
   without touching individual mocks (chapter 5).
2. **Endpoint match.** Mockxy looks among the registered routes for the one matching the
   request path — the most specific always wins — and then checks that the route defines
   the request's HTTP method. The precise rules (path parameters, declared queries,
   precedence) are in [chapter 7](07-creating-endpoints.md).
3. **The selected variant answers.** If the endpoint exists and is enabled, its selected
   variant answers: the static mock, the handler, or the middleware (which still goes
   through the backend). An active variant sequence may automatically choose which step to
   serve (chapter 12).
4. **Fallback.** If no mock covers the request, the **proxy fallback** decides: enabled
   (the default) and with a backend configured, the request is forwarded to the real
   backend; disabled, the answer is a `404 Mock Not Found` with the reason for the missed
   match in the body; enabled but with no backend configured, `501 Backend Not
   Configured`.

Fallback off is the **mock-only mode**: useful for offline demos, or to be certain the
frontend never touches a real environment. Chapter 26 goes deeper into fallback, errors
and timeouts.

## `x-mock-source`: who really answered

Every response leaving Mockxy carries a technical header, **`x-mock-source`**, declaring
who produced it. It is the number-one diagnostic tool — when a request doesn't do what you
expect, the first question is always "who answered?", and the answer is already in the
headers:

| Value | Meaning |
|---|---|
| `mock` | static response from a mock |
| `handler` | generated by a local handler |
| `middleware` | backend response transformed by a middleware |
| `backend` | proxied to the backend with no transformation |
| `mock-only` | `404` with proxy fallback disabled |
| `backend-unconfigured` | `501` because no backend is configured |
| `cors-preflight` | preflight `OPTIONS` handled by automatic CORS |

To read it from the browser: DevTools → Network tab → click the request → *Response
Headers* section. The same values appear as the "Served by" column in the monitor, which
is effectively this header's legend.

## The concepts on screen

To anchor the terms to the interface: in the catalog, each row is an **endpoint** (method
badge + path), grouped into **collections**; selecting one, the panel on the right shows
its **response variants**, with the selected one indicated. Everything you see corresponds
to files inside the open **workspace**.

> visually.
